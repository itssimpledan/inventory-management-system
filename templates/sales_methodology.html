1. Atome (BNPL Payment Gateway)

Atome is a Buy Now Pay Later (BNPL) gateway. It collects payment from the customer in instalments and disburses funds to the merchant minus its fees. Two reports are exported from the Atome merchant portal each month:

Transaction File
Filename prefix: transactions_
One row per customer transaction (sale or refund). Records revenue, GST, and whether the amount is still receivable (CONFIRMED) or already settled (FULLY_SETTLED).
Payment File
Filename prefix: payout_list_
One row per payout batch disbursed to our bank account. Records gross sales in that batch, all Atome fees (MDR + flat), sponsored vouchers, rebates, and net cash received.

Calculation Steps
#Field ProducedFormulaPurpose
1 GST on Revenue tx_amount × (9 ÷ 109) Extract GST portion from the GST-inclusive transaction amount
2 Revenue (ex-GST) tx_amount − gst_on_revenue Net revenue before GST — the P&L revenue line
3 Expenses (MDR + Flat) abs(mdr_fee) + abs(flat_fee) Total platform fees charged (already ex-GST in Atome's file)
4 AR Balance amt_receivable where status = CONFIRMED Amount still owed by Atome — booked to Accounts Receivable
5 Cash Received amt_receivable where status = FULLY_SETTLED Cash already disbursed to our bank account
6 Brand Split Derived from E-commerce Platform column mapping Allocates revenue to AndSons or MM brand for management reporting
Accounting Entries — per Transaction Row
Revenue Recognition Entry
DR Accounts Receivable (Atome) tx_amount
CR Revenue (ex-GST) revenue_ex_gst
CR GST Payable gst_on_revenue
Booked for every MERCHANT_PAYMENT row regardless of settlement status.
Platform Fee Entry
DR Platform Fees Expense expenses_total
CR Accounts Receivable (Atome) expenses_total
Atome deducts fees from the AR balance. The net AR is what gets settled in the payment file.
Note on COGS: The Atome transaction file records how a customer paid — not what they bought. COGS must be calculated from the corresponding order export (Shopify, Lazada, etc.) using SKU-level line items matched to this period's FIFO unit cost.
{# ── 1b. Payment File ─────────────────────────────────────────────────── #}
1b. Atome Payment File payout_list_ export
Source Columns Used
Column HeaderDescriptionSign Convention
Payout IDUnique identifier for this disbursement batch—
Payout DateDate the batch was disbursed to bank—
Payout Amount (E+F+G+H)Net cash transferred to our bank accountPositive = cash in
Total Sales (E)Gross sales included in this payout batch (incl. GST)Positive
Total Sponsored Voucher Amount (F)Voucher subsidy adjustments by AtomeUsually negative (reduces payout)
All Atome Fees (G)MDR + flat fees charged on this batchNegative (deducted from payout)
All Rebates (H)Rebates returned to merchant by AtomePositive (adds to payout)
Arithmetic identity: E + F + G + H = Payout Amount. The system flags rows where the variance exceeds $0.02 (rounding tolerance).
Calculation Steps
#Field ProducedFormulaPurpose
1 Fees Absolute abs(All Atome Fees) Convert negative fee figure to positive for accounting debit
2 Fees (ex-GST) fees_abs ÷ 1.09 Back-calculate net fee before 9% GST (GST-inclusive rate: 9 ÷ 109)
3 GST on Fees fees_abs − fees_ex_gst GST component claimable as input tax credit
4 Arithmetic Check |E + F + G + H − Payout| ≤ 0.02 Validates file integrity; flags row if mismatch exceeds $0.02
Accounting Entries — per Payout Batch Row
Cash Settlement Entry (clears AR from Transaction File)
DR Cash / Bank Payout Amount Net cash received
DR Platform Fees Expense (ex-GST) fees_ex_gst Fee before GST
DR GST Input Tax (on Fees) gst_on_fees Claimable input tax credit (9/109 of fees)
DR Sponsored Voucher Expense |sponsored_voucher| If negative in file (reduces payout)
CR Accounts Receivable (Atome) Total Sales Clears the AR raised by the transaction file
CR Rebate Income rebates If positive in file (increases payout)
Check: Total DR = Total CR. Any imbalance indicates a variance that requires investigation (see Reconciliation section).
Timing Note: Atome may include transactions from prior months in a payout batch. The payment file should be posted using the payout date as the accounting date, not the original transaction date. Differences in timing between the transaction file and payment file are normal and are reconciled via the AR account.
{# ── 1c. Reconciliation Checklist ────────────────────────────────────────#}
1c. Month-End Reconciliation Checklist
#StepHow to Verify
1 Upload both files for the month Sales → Upload Sales File. Upload Transaction file first, then Payment file.
2 Check for flagged rows in each upload Any flagged rows appear in red on the detail page. Investigate and resolve before posting.
3 AR variance: confirm Transaction AR ≈ Payment Total Sales Sum of CONFIRMED amt_receivable in transaction file should reconcile to Total Sales column in payment file (timing differences are expected — track them in the AR sub-ledger).
4 Verify fee GST calculation fees_ex_gst × 1.09 should equal fees_incl_gst (within $0.01 rounding). System flags mismatches automatically.
5 Post journal entries in accounting system Use the indicative entries shown on each upload detail page. Post by brand for management reporting.
6 Upload corresponding order file for COGS Upload the Shopify / platform order export for the same period to calculate cost of sales via FIFO.

3. GST Quick Reference
Extract GST from Inclusive Price
GST = amount × (9 ÷ 109)
When amount includes GST (e.g. transaction amount)
Add GST to Ex-GST Price
Incl. GST = amount × 1.09
When amount is before GST
Back-Calculate Ex-GST from Inclusive
Ex-GST = amount ÷ 1.09
Used for Atome fee GST extraction
Singapore GST Rate: 9% (effective 1 Jan 2024). All calculations in this system use 9% GST. Update this SOP and the parser code if the rate changes.
